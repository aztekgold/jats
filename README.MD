# Agentable v1.0.0 Specification
## The JSON Agentic-Friendly Table Standard

Agentable is a platform-agnostic, agent-friendly JSON format for structured data tables. It embeds meaning, structure, and permissions directly into tables, allowing agents to safely read, write, and exchange data with minimal overhead. Agentable is both human- and machine-readable, designed for clarity, longevity, and portability.

## 1. Introduction

AI agents can read tables, but they cannot reliably understand them. Formats like CSV, Markdown, and plain JSON store values but not meaning. They lack typing, context, identity, and safe interaction boundaries, forcing agents to guess what data represents and making joins unsafe, automation brittle, and reasoning error-prone.

AGENTABLE addresses this problem by providing a standardized, deterministic table schema with semantic anchors (field descriptions), a strict typing system, constraints, and a collision-resistant ID strategy. This enables distributed, multi-agent table writing without coordination, making structured data safer, clearer, and more reliable for both humans and AI.

## 2. Design Philosophy

*   **Platform Agnostic:** Portable JSON structure that can live in a database, a local file, or an API response.
*   **Actor-Aware:** Explicit `allowAgent` policies define the machine's boundaries, ensuring safe interaction.
*   **Sortable Unique ID:** Unified ID generation logic ensures data is naturally sortable and unique without central counters.
*   **Semantic Anchor:** In-line descriptions provide "Decision Support" for AI Agents, allowing LLMs to understand the *meaning* of a column, not just its key.

## 3. Unified ID Generation

AGENTABLE uses **Base36** encoding for all identifiers to balance token-friendly compression with naturally sortable, URL-friendly, and case-safe strings. IDs are always stored and compared in **lowercase**; implementations that produce uppercase Base36 (e.g. some languages' built-in encoders) must normalize to lowercase before use.

### 3.1 Schema IDs (Prefix + 3 Random)
Short, stable handles for structural elements.
*   **Columns:** `col_` + 3 random chars (e.g., `col_x9z`)
*   **Views:** `view_` + 3 random chars (e.g., `view_b2y`)
*   **Filters:** `flt_` + 3 random chars (e.g., `flt_k5p`)
*   **Sorts:** `srt_` + 3 random chars (e.g., `srt_w1r`)

### 3.2 Row IDs (9 Time + 3 Random)
A 12-character string: `TTTTTTTTTRRR`.
*   **Time (9 chars):** `Date.now()` encoded in Base36.
*   **Random (3 chars):** Suffix generated by the same `random3Char()` utility.
*   **Logic:** This allows approx. 46 unique IDs per millisecond per node, ensuring perfect chronological sorting and collision resistance suitable for high-frequency agentic writes.

## 4. Schema Definition

The AGENTABLE schema is defined using TypeScript interfaces below, but is language-agnostic in practice.

### 4.1 Columns

Columns define the structure and constraints of the data.

```typescript
type AgentableColumnType = 
  | "text" 
  | "number" 
  | "select" 
  | "date" 
  | "boolean" 
  | "url"
  | "link";

interface AgentableOption {
  value: string;
  color?: string; // UI hint: "red", "#ff0000", etc.
}

interface AgentableColumn {
  id: `col_${string}`;
  name: string;
  type: AgentableColumnType;
  description?: string; // Context for Agents
  display: {
    width: number;
    dateFormat?: string; // Optional: UI format (e.g. "YYYY-MM-DD")
  };
  constraints?: {
    multiSelect?: boolean;
    options?: AgentableOption[];
    required?: boolean;
    min?: number;
    max?: number;
    pattern?: string;
  };
}
```

### 4.2 Views

Views correspond to saved states of the table, including filtering, sorting, and visibility.

```typescript
interface AgentableFilter {
  id: `flt_${string}`;
  columnId: string;
  operator: "is" | "isNot" | "contains" | "startsWith" | "endsWith" | "gt" | "lt" | "isEmpty" | "isNotEmpty";
  value: any;
}

interface AgentableSort {
  id: `srt_${string}`;
  columnId: string;
  direction: "asc" | "desc";
}

interface AgentableView {
  id: `view_${string}`;
  name: string;
  description: string;
  filters: AgentableFilter[];
  sorts: AgentableSort[];
  hiddenColumns: string[];   // List of column IDs to hide
  columnOrder: string[];     // List of column IDs in display order
}
```

### 4.3 Rows

Rows store the actual data. Cells are keyed by Column ID, not component name, to allow renaming columns without breaking data.

```typescript
interface AgentableRow {
  id: string; // 12 chars: 9 Time + 3 Random
  cells: Record<string, any>; // Keyed by col_id (e.g. "col_a1b": "Value")
}
```

### 4.4 Metadata & Policy

```typescript
interface AgentableSchema {
  version: "1.0.0";
  metadata: {
    title: string;
    description: string;
  };
  policy: {
    permissions: {
      allowAgentRead: boolean;
      allowAgentCreate: boolean;
      allowAgentUpdate: boolean;
      allowAgentDelete: boolean;
    };
  };
  columns: AgentableColumn[];
  views: AgentableView[];
  rows: AgentableRow[];
}
```

## 5. Helper Logic

Standardized logic for generating compliant IDs.

```typescript
/**
 * Generates the unified 3-character random suffix.
 */
function random3Char(): string {
  return Math.floor(Math.random() * 46656).toString(36).padStart(3, "0");
}

/**
 * AGENTABLE Row ID: 9 Chars Time (Base36) + 3 Chars Random
 */
export function generateRowId(): string {
  return Date.now().toString(36).padStart(9, "0") + random3Char();
}

/**
 * AGENTABLE Schema IDs
 */
export const generateColId = () => `col_${random3Char()}`;
export const generateViewId = () => `view_${random3Char()}`;
export const generateFilterId = () => `flt_${random3Char()}`;
export const generateSortId = () => `srt_${random3Char()}`;
```

## 6. Developer Libraries

AGENTABLE provides official implementations in **TypeScript** and **Python** to simplify table management, validation, and migration.

### 6.1 Manager Class
The `Manager` (TS) or `AgentableManager` (Py) is the primary entry point for manipulating tables.

**Key Features:**
*   **Automatic ID Generation:** Handles collision-protected IDs for columns, rows, views, filters, and sorts.
*   **Strict Validation:** Uses Zod (TS) or Pydantic (Py) to ensure every change remains schema-compliant.
*   **Minimalist API:** Focuses on surgical updates (e.g., `updateColumn`, `updateView`).

**Common Methods:**
*   `addColumn({ name, type, ... })`: Adds a new column.
*   `updateColumn(id, { ...updates })`: Rename columns or change types.
*   `addRow({ col_id: value })`: Adds a data row.
*   `moveRow(id, toIndex)`: Surgical row reordering.
*   `createView(name)`: Creates a new saved view.
*   `updateView(id, { name, columnOrder, ... })`: Update view metadata or visibility.
*   `setColumnVisibility(viewId, colId, visible)`: Easily toggle column hide/show.
*   `addFilter(viewId, { columnId, operator, value })`: Add a view filter.
*   `addSort(viewId, { columnId, direction })`: Add a view sort.
*   `updateMetadata({ title, description })`: Rename the table or update its context.

### 6.2 Usage (TypeScript)
```typescript
import { createTable } from "@aztekgold/agentable";

const table = createTable({ metadata: { title: "Task List" } });
const col = table.addColumn({ name: "Task", type: "text" });
table.addRow({ [col.id]: "Buy milk" });

const json = table.getAgentable(); // Get the clean JSON
```

### 6.3 Usage (Python)
```python
from agentable import create_table

table = create_table({"metadata": {"title": "Task List"}})
col = table.add_column(name="Task", type="text")
table.add_row({col.id: "Buy milk"})

data = table.to_dict() # Get the clean Dict
```

## 7. Agent Tooling Integrations (AGENTABLE Agent)

AGENTABLE provides native tooling features that allow backend systems to securely expose table operations directly to LLM agents. Features include:

*   **Native Zod Generation:** Dynamically builds precise parameter schemas strictly enforcing the `AgentableColumn` settings and configurations of the current table.
*   **Unified Provider Wrapping:** Tool definitions are shaped generically and can be cleanly exported natively to Vercel AI SDK, OpenAI, and Anthropic formatting (`toVercel()`, `toOpenAI()`, and `toAnthropic()`).
*   **Granular Boundary Protection:** Execution logic explicitly checks schema policy capabilities (`allowAgentCreate`, `allowAgentUpdate`, etc.) before fulfilling agent operations to maintain sandbox integrity.

The supported dynamic schema tools include:
1.  **`add_row`**: Dynamically maps necessary column insertions against defined columns.
2.  **`update_row`**: Enforces a strictly validated `row_id` field alongside safely mapped optional updates for the respective columns.
3.  **`add_select_option`**: Allows agents to systematically establish new distinct classification constraints directly on categorized `select` columns.

### 6.1 Permission Enforcement

Permissions defined in `policy.permissions` are **hard boundaries**. Agents MUST check the relevant permission flag before executing any operation.

| Operation | Required Permission |
|---|---|
| Reading table structure, rows, columns, views, filters, or sorts | `allowAgentRead` |
| Adding a new row, column, view, filter, or sort | `allowAgentCreate` |
| Updating a row, column, view, filter, or sort | `allowAgentUpdate` |
| Deleting a row, column, view, filter, or sort | `allowAgentDelete` |

**Rules:**
*   An agent MUST check the relevant permission flag **before** executing any operation.
*   An agent CAN NOT create, update, or delete rows, columns, views, filters, or sorts without the corresponding permission explicitly set to `true`.
*   An agent MUST NOT silently fail or partially execute a denied operation.
*   An agent MUST surface the denial to the caller with a structured error message explaining which permission was missing. The user may then grant access and the operation can be reattempted.
*   A missing permission flag (i.e. `undefined`) MUST be treated as `true` to ensure backwards compatibility with tables that predate explicit policy definitions.

**Denial Response Format:**

When a permission check fails, the tool MUST return a message in the following format:

```
Permission Denied: Agent is not allowed to [action] rows. Required permission: [permissionKey]
```

**Example:**
```
Permission Denied: Agent is not allowed to create rows. Required permission: allowAgentCreate
```
